# [답변] AwareFit 급여 정산 시스템 V4 명세서 기반 UI/UX 설계 및 코드 검토 결과

## 3.1. 코드 비교 및 영향도 분석

### 프론트엔드 코드 수정 필요 내역 (`src/components/staff/PayrollModalStaff.tsx` 기준)
1. **데이터 인터페이스 불일치:**
   - 기존: `classCount * 75000` 등 예상 고정 단가(7.5만원)에 단순히 총 수업 횟수(`staff.workHours`)를 곱하여 정산금(`classRevenue`, `classPay`)을 산출함.
   - V4 변경: `PayrollRecord` 및 `SessionLog` 배열 형식 데이터를 기준으로 개별 `unitPrice` × `commissionRate`를 건별로 합산해야 함.
2. **세금 계산 (과세표준) 로직 불일치:**
   - 기존: 단순히 `grossPay`를 기준으로 3.3% 또는 4대보험 비율을 일괄 적용함.
   - V4 변경: 세금을 산출할 때 `nonTaxableAllowance`(정규직 비과세수당)와 `refundClawback`(당월 환불 환수금)을 제외한 **'세금 계산용 과세표준'**을 먼저 산정하고 그 금액에 과세율을 적용하도록 로직 개선이 필요함. 최종 공제액(Total Deductions)에는 분리 계산된 세금과 환수금이 합산되어야 함.
3. **결제/승인 상태 관리 부재:**
   - 기존: `PENDING / APPROVED / PAID` 상태 흐름 없이 정산 결과만 단방향으로 조회하는 화면이었음.
   - V4 변경: `PayrollRecord.status` 필드 정보를 UI에 바인딩하고, 관리자가 상태 변경을 관장하는 UI 버튼을 도입해야 함.

### `unitPrice` 수강권 고정(Freeze) 로직 처리 UI 제안
수업 결제/생성 시점에 단가가 고정(`Freeze`)되어야 하며, 기본 상품 가격이 추후 오르더라도 기 결제된 수강권의 1회당 정산 단가는 변하지 않아야 함.
- **수강권/회원 상세 화면:** 수강권 내역 표기 시 해당 수강권의 **1회당 단가(unitPrice)**를 텍스트 뱃지로 고정시켜, "이 회원과의 수업은 건당 X원 기준으로 정산됨"을 명확히 안내해야 함.
- **수정 불가 안내 표기:** `SessionLog` 정보 상세조회 등에서 관리자가 단가 수정을 시도할 경우, '결제 시점 확정 단가로 소급 적용 및 수정 불가능'이라는 Tooltip을 띄워 데이터 무결성을 보장해야 함.

## 3.2. UI/UX 설계 반영 (코드 레벨 적용 내역)

다음 4가지 핵심 요소를 프론트엔드 급여 명세서 UI(`PayrollModalStaff.tsx`)에 적용합니다.

1. **정산 근거 데이터 조회 (Audit Trail Modal):**
   - 수업 수수료 항목 옆에 [세션 상세 내역] 버튼을 배치하고, 클릭 시 하위 팝업(또는 테이블)을 통해 이번 달 정산에 편입된 모든 `sessionLogs`의 날짜, 담당 회원, 단가를 표출합니다.
2. **대타/노쇼 가시성 및 귀속 비율 안내:**
   - Audit Trail 테이블 안의 각 열에 "정상", "노쇼(50% 인정)", "대타 진행(SPLIT 50%)" 등의 상태 태그 배지를 달아 명세서의 투명성을 극대화합니다.
3. **상태 흐름 관리 (Status Flow Indicator):**
   - 모달 상단 헤더에 "현재 상태: [정산 대기중 / 승인 완료 / 지급 완료]" 배지를 표시합니다.
   - 화면 하단에는 관리자 전용 "이번 달 명세서 확정(APPROVE)" 버튼을 활성화하여 클릭 시 파이프라인의 다음 단계로 진행할 수 있도록 유도합니다.
4. **환불 환수금 (Clawback) 툴팁 및 명시적 표현:**
   - 공제(Deductions) 섹션 하단에 사업소득세와 별도로 **"기수당 환수금"**을 명시합니다. 
   - 세금 툴팁에 **"과세표준 산출식: (총지급액 - 비과세수당 - 환수금) × 세율"**을 표기하여 무리하게 세금이 많이 청구된 것으로 의심받지 않도록 신뢰성 있는 전개식을 제공합니다.
